# Java 代码规范自动化检查方案

> 按场景组织，每个场景讲清楚问题、可选方案、选型理由和配置。

---

## 三层体系

代码规范分三层，每层解决不同层面的问题：

| 层级 | 解决什么问题 | 反馈速度 |
|------|-------------|:--------:|
| **格式化** | 缩进、空格、换行、import 排序——机器全权负责，零争议 | 亚秒级 |
| **规范检查** | 命名、Javadoc、代码结构——有明确标准但需要人遵守 | 实时 |
| **质量检查** | NPE、资源泄漏、性能隐患——机器辅助发现，人判断 | 分钟级 |

## 场景一：统一代码格式

团队多人多 IDE，缩进用 2 格还是 4 格？花括号是否换行？import 顺序怎么排？每次提交都有人改格式，合并冲突不断。

**思路**：格式化引擎 + 编排器，自动执行，人不参与争论。

### 格式化引擎选型

| 引擎 | 可配置性 | 输出确定性 | 适用场景 |
|------|:--------:|:---------:|---------|
| **google-java-format** | **零配置**（仅 Google/AOSP 两档） | ✅ 同一输入永远同一输出 | 新项目，或团队接受 Google 风格 |
| palantir-java-format | 低（可配置行宽、缩进） | ✅ | 需要调整行宽或缩进时替代 google-java-format |
| Eclipse Formatter | 极高（XML 配置上百项） | ❌ 不同版本可能不同 | 遗留项目绑定 Eclipse，无法换引擎 |

**推荐 google-java-format（默认）**：零配置，团队不需要争论"缩进 2 格还是 4 格"。大量开源项目（Guava、gRPC）在用，新人很可能已经熟悉。

如果默认风格无法满足：
- 需要 **4 空格缩进** → 切 AOSP 风格：`<style>AOSP</style>`
- 需要 **自定义行宽**（默认 100 字符）→ 考虑 `palantir-java-format`
- 需要 **完全自定义** → Eclipse Formatter，但要做好版本锁定

### 编排器选型

编排器把格式化引擎和 import 整理、空白清理等任务串起来。**推荐 Spotless**，核心理由是 `ratchetFrom`——只检查与 `origin/main` 有差异的文件，老代码不动，新代码必须干净。这是存量项目落地的核心能力。

### 配置

```xml
<plugin>
  <groupId>com.diffplug.spotless</groupId>
  <artifactId>spotless-maven-plugin</artifactId>
  <version>2.44.3</version>
  <configuration>
    <ratchetFrom>origin/main</ratchetFrom>
    <java>
      <googleJavaFormat>
        <!-- <style>AOSP</style> -->
      </googleJavaFormat>
      <importOrder>
        <order>java,javax,org,com,</order>
      </importOrder>
      <removeUnusedImports />
      <trimTrailingWhitespace />
      <endWithNewline />
    </java>
    <excludes>
      <exclude>build/**</exclude>
      <exclude>generated/**</exclude>
    </excludes>
  </configuration>
</plugin>
```

```bash
mvn spotless:apply   # 格式化所有改动
mvn spotless:check   # CI 用，只检查不改
```

### IDE 配合

- **IntelliJ**：安装 google-java-format 插件 → 保存时自动格式化
- **VS Code**：Extension Pack for Java + 配置 `"editor.formatOnSave": true`
- 保存 `.vscode/settings.json` 和 `.idea/codeStyles/` 到版本控制，新人 clone 即用

## 场景二：规范命名和注释

代码里出现 `UserDo`、`getuserinfo`、`a`、`b` 这类命名，方法没有 Javadoc，改代码全靠猜。

**思路**：Checkstyle 检查命名规范和 Javadoc，配合 Spotless 但不重复覆盖。PMD 也有 codestyle 规则，但 Checkstyle 的命名检查更精确。**每个规则只在一个工具中配置**，避免重复报警。

### 配置

```xml
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-checkstyle-plugin</artifactId>
  <version>3.6.0</version>
  <configuration>
    <configLocation>checkstyle.xml</configLocation>
    <failOnViolation>true</failOnViolation>
    <violationSeverity>warning</violationSeverity>
    <excludes>**/generated/**/*</excludes>
  </configuration>
  <executions>
    <execution>
      <phase>verify</phase>
      <goals><goal>check</goal></goals>
    </execution>
  </executions>
</plugin>
```

```xml
<!-- checkstyle.xml -->
<module name="Checker">
  <property name="charset" value="UTF-8"/>
  <property name="fileExtensions" value="java"/>
  <module name="TreeWalker">
    <!-- 命名规范 -->
    <module name="PackageName"/><module name="TypeName"/>
    <module name="MethodName"/><module name="ParameterName"/>
    <module name="LocalVariableName"/><module name="ConstantName"/>
    <module name="MemberName"/>
    <!-- Javadoc -->
    <module name="JavadocMethod"/><module name="JavadocType"/>
    <module name="JavadocStyle"/>
    <!-- 关键 bug 预防 -->
    <module name="EqualsHashCode"/>
    <module name="StringLiteralEquality"/>
    <module name="SimplifyBooleanExpression"/>
    <module name="SimplifyBooleanReturn"/>
  </module>
</module>
```

### 注意事项

- **Javadoc 范围**：内部项目可在 `JavadocMethod` 上设 `scope="public"`，只要求公开方法写注释
- **Lombok 兼容**：`@Data`、`@Value` 注解的类需在 `suppressions.xml` 中排除 Javadoc 检查
- **不做重复检查**：格式化的事（缩进、import、花括号）不归 Checkstyle 管。`UnusedImports` 规则建议关闭——Spotless 的 `removeUnusedImports` 更准确

### IDE 配合

**VS Code** 安装 **Alibaba Java Coding Guidelines** 插件，写代码时实时红线提示命名、Javadoc 等违规。配合 **Error Lens** 将警告信息直接显示在行尾，不用悬停鼠标查看。无需等 CI 跑完就能发现规范问题。

## 场景三：发现潜在 Bug

线上出现 NPE、资源未关闭、equals 误用、空循环等低级错误，人工 CR 很难全发现。

**思路**：PMD 做静态分析，自动扫描高风险代码模式。

### 为什么选 PMD 而不是其他？

| 工具 | 分析方式 | 优势 | 劣势 |
|------|---------|------|------|
| **PMD** | 源码 AST | 不需编译，速度快，规则易懂 | 跨文件分析弱 |
| SpotBugs | 字节码 | 能发现深层 bug（如复杂 NPE 路径） | 需先编译，CI 慢 |
| Error Prone | 编译插件 | 编译期直接拦截 | 侵入性强，规则较少 |

**PMD 起步**，后续可补充 SpotBugs，两者不冲突。

### 规则开启策略

按信号质量分级，分阶段开启，避免噪音淹没真实问题：

| 类别 | 信号质量 | 建议 | 典型规则 |
|------|:-------:|------|---------|
| `errorprone` | ⭐⭐⭐ | **先开** | `CloseResource`、`EmptyCatchBlock`、`EqualsNull` |
| `bestpractices` | ⭐⭐⭐ | **先开** | `SwitchStmtsShouldHaveDefault`、`AvoidPrintStackTrace` |
| `security` | ⭐⭐⭐ | **先开** | `HardCodedCryptoKey`、`InsecureCryptoIv` |
| `performance` | ⭐⭐ | 适应后开 | `ConsecutiveLiteralAppends`、`InefficientStringBuffering` |
| `codestyle` | ⭐ | 选择性开 | 不与 Checkstyle 重叠的规则 |
| `design` | ⭐ | 可选 | 存量项目噪音极大 |

### 配置

```xml
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-pmd-plugin</artifactId>
  <version>3.26.0</version>
  <configuration>
    <failOnViolation>true</failOnViolation>
    <rulesets><ruleset>pmd-rules.xml</ruleset></rulesets>
    <excludes><exclude>**/generated/**</exclude></excludes>
  </configuration>
  <executions>
    <execution>
      <phase>verify</phase>
      <goals><goal>check</goal></goals>
    </execution>
  </executions>
</plugin>
```

```xml
<!-- pmd-rules.xml -->
<ruleset name="Custom Rules"
  xmlns="http://pmd.sourceforge.net/ruleset/2.0.0"
  xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
  xsi:schemaLocation="http://pmd.sourceforge.net/ruleset/2.0.0 https://pmd.sourceforge.net/ruleset_2_0_0.xsd">
  <rule ref="category/java/errorprone.xml">
    <exclude name="DataflowAnomalyAnalysis"/>  <!-- 误报率高 -->
    <exclude name="BeanMembersShouldSerialize"/>
  </rule>
  <rule ref="category/java/bestpractices.xml">
    <exclude name="JUnitAssertionsShouldIncludeMessage"/>
    <exclude name="JUnitTestContainsTooManyAsserts"/>
  </rule>
  <rule ref="category/java/security.xml"/>
  <rule ref="category/java/performance.xml"/>
  <rule ref="category/java/codestyle.xml/UnnecessaryImport"/>
  <rule ref="category/java/codestyle.xml/UnnecessaryModifier"/>
  <rule ref="category/java/codestyle.xml/UnnecessaryReturn"/>
</ruleset>
```

```bash
mvn pmd:check   # 仅 PMD
mvn verify      # 全量（格式化 → 规范 → 质量）
```

## 场景四：存量项目渐进落地

项目跑了 3 年、几千个文件。全量开启检查会爆出数万个违规，一次性修不完。

**思路**：分五阶段推进——新代码必须干净，老代码逐步洗白。

| 阶段 | 时间 | 操作 | 验收标准 |
|------|:----:|------|---------|
| **① 格式化** | 第 1 周 | 只开 Spotless + `ratchetFrom` | 无人抱怨格式化打断开发 |
| **② 规范检查** | 第 2-4 周 | 开 Checkstyle，`failOnViolation=false`，修 Top-10 | 违规趋势下降 |
| **③ 质量检查** | 第 2 个月 | 开 PMD `errorprone` + `bestpractices`，修真实 bug | 至少发现 3 个隐患 |
| **④ 锁死** | 第 3 个月 | `failOnViolation=true`，CI 强制通过 | 所有 PR 走检查 |
| **⑤ 全量格式化** | 时机成熟 | `mvn spotless:apply` + `.git-blame-ignore-revs` | blame 不受影响 |

### `ratchetFrom` 原理

`ratchetFrom=origin/main` 本质是 `git diff origin/main --name-only` + Spotless 格式化，只扫改动的文件。存量代码不动，零风险。**CI 必须 `fetch-depth: 0`（完整 clone）**，否则 ratchet 无法工作。

### 抑制机制

对于确实需要豁免的老文件或特殊情况：

```xml
<!-- suppressions.xml — Checkstyle 豁免 -->
<suppressions>
  <suppress files="legacy/.*" checks="JavadocMethod"/>
  <suppress files="generated/.*" checks=".*"/>
</suppressions>
```

```java
// 行级抑制，旁边必须有原因注释
@SuppressWarnings("PMD.CloseResource")
public void legacyMethod() {
    // TODO(PROJ-123): 重构后移除
}
```

## 场景五：CI 自动检查

本地忘了跑检查、IDE 没装插件——混乱代码被合并到主分支。

**思路**：CI 自动跑格式 + 规范 + 质量检查，不通过不能合并。

```yaml
`.github/workflows/lint.yml`
name: Code Quality
on: [push, pull_request]

jobs:
  format:
    name: Format Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0   # ratchetFrom 需要完整 git 历史
      - uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
          cache: maven
      - run: mvn spotless:check

  lint:
    name: Lint Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
          cache: maven
      - run: mvn verify
```

**要点**：`format` 和 `lint` 拆成两个 job——失败时一眼知道是格式还是质量问题，也支持并行跑。分支保护中 `lint` 初期可设为 optional，`format` 必须过。

## 命令速查

```bash
mvn spotless:apply      # 格式化
mvn spotless:check      # 检查格式
mvn checkstyle:check    # 规范检查
mvn pmd:check           # 质量检查
mvn verify              # 全量（格式 → 规范 → 质量）
```

## 职责边界

| 关注点 | 负责工具 | 选型理由 |
|--------|---------|---------|
| 缩进、换行、空格 | **google-java-format** (via Spotless) | 零配置，不接受争论 |
| import 排序 | **Spotless** `importOrder` | 编排器统一管理 |
| 清理无用 import | **Spotless** `removeUnusedImports` | 比 Checkstyle/PMD 准确 |
| 命名规范 | **Checkstyle** | 精确、可读性好 |
| Javadoc | **Checkstyle** | 原生支持 |
| 潜在 bug（NPE、资源） | **PMD** `errorprone` | 源码级，不需编译 |
| 安全漏洞 | **PMD** `security` | 硬编码密码、SQL 注入 |
